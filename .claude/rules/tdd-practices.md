# TDD Practices — 汎用テスト駆動開発の心得

テスト駆動開発の心得。全開発エージェントが遵守すること。

---

## 原則: Red → Green → Refactor

```
1. RED    — 失敗するテストを先に書く
2. GREEN  — テストを通す最小限の実装を書く
3. REFACTOR — テストが通る状態を維持しつつコードを改善する
```

**このサイクルを崩してはならない。** 実装を先に書いてから「後でテストを書く」は TDD ではない。

---

## テストピラミッド

```
        ╱ E2E ╲                        ← 少数・遅い・高信頼
       ╱───────╲
      ╱  Integration  ╲               ← 中程度
     ╱─────────────────╲
    ╱    Unit Tests      ╲            ← 多数・速い・安価
   ╱───────────────────────╲
```

| レイヤー | 対象 | 目安比率 |
|---------|------|---------|
| Unit | ユーティリティ関数、パーサー、変換ロジック | 70% |
| Integration | API エンドポイント、ミドルウェア連携 | 20% |
| E2E | クリティカルパス（主要ユーザーフロー） | 10% |

具体的なテストフレームワークはプロジェクトの CLAUDE.md または tech-stack.md で定義する。

## ユニットテストの書き方

### DO

| # | ルール |
|---|--------|
| U1 | **1テスト1アサーション**を原則とする。複数のアサーションが必要な場合はテストを分割する |
| U2 | テスト名は **「何が」「どうなるか」** を明記する。`it('returns empty array when directory is empty')` |
| U3 | **Arrange → Act → Assert** パターンで構造化する |
| U4 | 境界値を必ずテストする（0, 1, MAX, 空文字列, null, undefined） |
| U5 | エラーケースをテストする（不正入力、存在しないパス、権限エラー） |
| U6 | テストは **独立**して実行できること。テスト間の順序依存を作らない |
| U7 | 外部依存（ファイルシステム、ネットワーク）はモック化する |

### DON'T

| # | 禁止 | 理由 |
|---|------|------|
| X1 | 実装の内部詳細をテストする | リファクタリングでテストが壊れる |
| X2 | テストにロジックを入れる（if/for/try-catch） | テスト自体にバグが入る |
| X3 | `test.skip` を放置する | スキップされたテストは存在しないのと同じ |
| X4 | スナップショットテストに頼りすぎる | 変更のたびに更新が必要で意味が薄れる |

## テストカバレッジ基準

| 対象 | 目標 |
|------|------|
| ユーティリティ関数 | 90% |
| ルートハンドラ / コントローラ | 80% |
| ミドルウェア | 80% |
| クライアントサイド | 60%（DOM依存部分は E2E でカバー） |
| 全体 | 75% |

## 心得

1. **テストは仕様書である。** コードを読まなくても、テストを読めば機能が分かる状態を目指す
2. **テストが書きにくいコードは設計が悪い。** テスタビリティの低さは設計改善のシグナル
3. **壊れたテストは即座に直す。** `test.skip` で逃げない
4. **テストの実行速度を守る。** Unit + Integration は3分以内、E2E は5分以内
5. **リファクタリングの安全網としてテストを使う。** テストがあるからこそ、大胆にリファクタリングできる
